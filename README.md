Техническое задание (ТЗ)
Маркетплейс коллекционных карт «CardMarket»
1. Цель и назначение проекта
Цель: Разработка платформы для купли-продажи и обмена коллекционными картами (например, спортивные карточки, Magic: The Gathering, Pokemon TCG) с функционалом аукционов и верификации подлинности.
Назначение: Предоставить пользователям безопасное и функциональное пространство для торговли, исключающее мошенничество за счет модерации и системы проверки.
2. Функциональные требования (Основные модули)
2.1. Пользовательский модуль (Auth & Profiles)
    • Регистрация и авторизация: По email/паролю и через OAuth (Google).
    • Роли:
        ◦ Гость: Просмотр лотов.
        ◦ Продавец/Коллекционер: Создание лотов, участие в торгах.
        ◦ Верифицированный продавец: Имеет значок "Check", доступ к расширенным лимитам.
        ◦ Администратор/Модератор: Управление пользователями, модерация карт и аукционов.
    • Профиль пользователя: Аватар, рейтинг (отзывы), статистика продаж, список избранного, история ставок.
2.2. Модуль карточек (Каталог)
    • Создание лота (карточки): Форма с полями: название, год выпуска, издательство, состояние (Mint, Excellent, Good), описание, цена (для обычной продажи) или стартовая цена (для аукциона).
    • Загрузка медиа: Загрузка нескольких фото карты (до 10 шт.). Требование: фото должно быть четким, без бликов.
    • Поиск и фильтрация:
        ◦ По названию, категории, состоянию, цене.
        ◦ По статусу: "Аукцион", "Купить сейчас", "Только с верификацией".
2.3. Модуль верификации (Экспертная оценка)
    • Запрос на верификацию: Продавец может подать заявку на проверку подлинности карты перед продажей.
    • Процесс модерации (бэк-офис):
        ◦ Модератор получает задачу.
        ◦ Сравнивает фото карты с эталонными базами (или использует свой опыт).
        ◦ Ставит статус: Верифицирована (с присвоением уникального номера сертификата) или Отклонена (с указанием причины).
    • Отображение: Верифицированные карты имеют специальный бейдж (галочку) в каталоге и детальной карточке.
2.4. Модуль аукционов (Realtime Bidding)
    • Создание аукциона: Продавец указывает дату и время окончания торгов, минимальный шаг ставки, опцию "Блиц-цена" (выкуп сейчас).
    • Процесс торгов:
        ◦ Участники видят текущую ставку, историю ставок и таймер окончания.
        ◦ Система автоматически уведомляет пользователя, если его перебили.
    • Realtime механики:
        ◦ При окончании аукциона система определяет победителя и автоматически создает черновик заказа.
        ◦ Если ставка сделана в последние 2 минуты (Anti-sniping), время аукциона продлевается еще на 2 минуты.
2.5. Модуль модерации фото
    • Автоматическая пре-модерация (AI-assisted): При загрузке фото, бэкенд (NodeJS) отправляет изображение на проверку (через внешние API типа AWS Rekognition или Google Vision) на предмет запрещенного контента.
    • Ручная модерация: Если AI не может определить качество, фото попадает в очередь модератора.
    • Требования к качеству: Изображение должно быть не менее 800x800px, не размытое, карта должна занимать не менее 70% кадра.
3. Технические требования (Стек и архитектура)
3.1. Бэкенд (NodeJS)
    • Фреймворк: NestJS (рекомендуется для сложной архитектуры) или Express.js + TypeScript.
    • База данных:
        ◦ PostgreSQL: Для хранения пользователей, карточек, заказов (реляционные данные).
        ◦ Redis: Для кэширования (топ-лотов), хранения сессий и, главное, для брокера сообщений в аукционах (Pub/Sub).
    • Realtime (Торги): WebSockets (Socket.io) или Server-Sent Events (SSE). Сервер должен отправлять обновления ставок всем подключенным клиентам к конкретному аукциону.
    • Очереди задач (Background Jobs):
        ◦ Использование BullMQ (на базе Redis) для обработки тяжелых задач:
            1. Отправка email-уведомлений о подтверждении ставок.
            2. Обработка завершившихся аукционов (начисление баллов, создание заказов).
            3. Запуск задач фото-модерации.
    • Фото-модерация: Интеграция с API провайдерами (например, AWS Rekognition для выявления неприемлемого контента или Google Vision для определения качества изображения). Файлы фото хранятся в объектном хранилище (S3 / MinIO).
3.2. Фронтенд (ReactJS)
    • Язык: TypeScript.
    • Стейт-менеджмент: Redux Toolkit (с RTK Query) или MobX.
    • Компонентный подход: Использование библиотек компонентов (Material-UI, Ant Design или Chakra UI) для ускорения разработки интерфейса модерации и каталога.
    • Realtime клиент: Подключение к WebSocket серверу для обновления цен в реальном времени на странице аукциона.
4. Требования к базам данных (Ключевые сущности)
    1. Users: id, email, passwordHash, role, isVerified (как продавец), rating.
    2. Cards: id, title, description, year, condition, images[ ], status (draft, moderation, active, sold), price, sellerId.
    3. Auctions: id, cardId, startPrice, currentPrice, minBidStep, startTime, endTime, status (active, closed, cancelled).
    4. Bids: id, auctionId, userId, amount, createdAt.
    5. VerificationRequests: id, cardId, moderatorId, status (pending, approved, rejected), certificateNumber, comment.
5. Алгоритмы и логика работы (Use Cases)
5.1. Сценарий "Создание карточки с аукционом"
    1. Пользователь (Продавец) заполняет форму на фронтенде (React).
    2. Фронтенд отправляет multipart/form-data (данные + фото) на бэкенд NodeJS.
    3. Бэкенд:
        ◦ Сохраняет фото во временное хранилище.
        ◦ Создает запись в БД Cards со статусом moderation.
        ◦ Ставит задачу в очередь (BullMQ) на фото-модерацию.
    4. Воркер фото-модерации берет задачу, прогоняет фото через AI-сервис.
        ◦ Если ОК: Статус карточки меняется на active, создается запись в Auctions.
        ◦ Если Fail: Статус rejected, продавец получает уведомление.
5.2. Сценарий "Реал-тайм торги"
    1. Клиент A и Клиент B открыли страницу аукциона. Они подключены к комнате auction:123 через WebSocket.
    2. Клиент B ставит ставку. Фронтенд отправляет запрос POST /api/auctions/123/bid.
    3. Бэкенд проверяет: сумма больше текущей, время не истекло.
    4. Если ОК: Бэкенд сохраняет ставку в БД, обновляет currentPrice в Auctions, и публикует событие newBid в Redis Pub/Sub.
    5. WebSocket Gateway слушает Redis и рассылает сообщение всем клиентам в комнате auction:123 (обновление цены и ленты активности).
6. Требования к интерфейсу (UI/UX)
    • Каталог: Сетка (Grid) с карточками товаров. Крупные превью.
    • Детальная карточка лота: Крупное фото (лупа для zoom), блок с информацией о верификации, история ставок (лента), форма ввода ставки.
    • Личный кабинет модератора: "Лента" задач на модерацию с возможностью пролистывать фото и нажимать "Подтвердить/Отклонить" без перезагрузки страницы (использовать React Query для оптимизации).
7. Требования к безопасности
    1. JWT токены: Access и Refresh токены для авторизации.
    2. Валидация: Тщательная валидация входных данных на бэкенде (классы DTO с декораторами class-validator).
    3. Защита ставок: Проверка, что пользователь не может поставить ставку выше своего баланса (если используется внутренний кошелек). Защита от накрутки ставок самим продавцом (запрет на ставки от владельца лота).
    4. Ограничение скорости (Rate Limiting): На эндпоинты создания ставок (не более 1 ставки в 3 секунды) для защиты от DDOS и флуда.
8. Этапы разработки (Roadmap)
    1. MVP (Minimum Viable Product): Регистрация, каталог, создание лотов с ручной модерацией фото, система обычной купли-продажи.
    2. Этап 2: Запуск аукционов с базовым WebSocket уведомлениями (без анти-снайпинга).
    3. Этап 3: Интеграция AI-модерации фото, система верификации карт экспертами, рейтинги.
    4. Этап 4: Внедрение системы продления аукциона (анти-снайпинг), push-уведомления, интеграция с платежными системами.
